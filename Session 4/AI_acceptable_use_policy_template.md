## Section 2 — Permitted Uses

*What AI tools may be used for in this organisation.*

The following uses are permitted within the data handling rules set out in Section 4:

- [ ] Drafting and editing internal documents, emails, and reports — where the content does not include restricted data
- [ ] Summarising publicly available information or approved internal documentation
- [ ] Generating ideas, outlines, or first drafts for review and editing by a human
- [ ] Analysing anonymised or aggregated data with no identifying information
- [ ] Searching and retrieving information from approved internal knowledge bases
- [ ] Automating routine tasks — such as formatting, classification, or routing — where a human reviews the output before action is taken

## Section 3 — Prohibited Uses

*What is explicitly not permitted.*

The following uses are prohibited under this policy, regardless of which AI tool is used:

**Data classification limits**

- [ ] Submitting data classified as Confidential, Restricted, or above to any AI tool without explicit written approval from IT
- [ ] Submitting personal data as defined under UK GDPR — including names, contact details, national insurance numbers, health data, or any combination of data that could identify an individual
- [ ] Submitting customer data of any classification
- [ ] Submitting financial data, including unreleased figures, forecasts, pricing models, or budget information
- [ ] Submitting intellectual property — including product designs, proprietary processes, source code, or contractual terms

**Behavioural prohibitions**

- [ ] Using AI tools to generate, modify, or distribute content that is misleading, discriminatory, or in violation of any applicable law
- [ ] Using AI tools to make automated decisions that affect individuals — hiring, performance review, access rights — without human review and sign-off
- [ ] Using AI tools to circumvent security controls, access controls, or data loss prevention policies
- [ ] Using personal consumer AI accounts to process work data

> **Plain English:** If you would need permission to share it externally by email, you need permission before submitting it to an AI tool.

*What data may and may not be submitted to external AI tools.*

Use the table below to record your organisation's data classification scheme and the corresponding AI tool rule for each level:

| Data Classification | Example Data Types | May be submitted to Approved AI tools? | Conditions / Restrictions |
|---|---|---|---|
| Public | Publicly available information, published documentation | Yes | None |
| Internal | General internal documents, meeting notes, non-sensitive reports | Yes | Approved tools only; no identifying information |
| Confidential | Customer data, financial forecasts, HR data, strategic plans | No | Written IT approval required in all cases |
| Restricted / Secret | Regulated data, legal privilege, board-level material | No | Prohibited without explicit written approval from CISO |

**If your organisation uses a different classification scheme**, update the rows above to match your internal framework.

**Third-party AI tools and data residency**

Before submitting any data to a third-party AI tool, confirm:

- [ ] The tool is on the Approved Tool List *(see Section 5)*
- [ ] The tool's data processing agreement (DPA) has been reviewed by IT or Legal
- [ ] Data does not leave the agreed geographic boundary *(e.g. UK/EU data residency requirements)*
- [ ] The tool does not use submitted data to train its models *(check the provider's terms)*

## Section 5 — Human Oversight Requirements

*Which AI-generated outputs require human review before acting on them.*

Not all AI outputs carry the same risk. Use this section to define where human review is mandatory before any action is taken.

**Human review is always required before:**

- [ ] Any AI-generated output that will be shared externally — with customers, partners, regulators, or the public
- [ ] Any AI-generated recommendation that will inform a business decision
- [ ] Any AI-generated output that affects an individual — performance, access rights, communications
- [ ] Any AI-generated summary of regulated, legal, or compliance-related content
- [ ] Any automated action taken by an AI agent on behalf of the organisation — file modification, message sending, system changes

## Section 6 — Incident Reporting

*What to do if an AI tool produces harmful, incorrect, or unexpected output — or if data has been submitted in error.*

**Report immediately if:**

- You have submitted data to an AI tool in violation of this policy
- An AI tool has behaved in an unexpected, harmful, or potentially malicious way — including producing outputs that appear to be manipulated by content in the prompt *(see: prompt injection)*
- You suspect an AI tool has accessed data or systems it was not intended to access
- An AI-generated output has been shared externally before human review was completed

**How to report:**

> Contact the IT Security team at: _________________________ [security team contact / email / internal ticket system]
>
> Include: what tool was used, what data was involved, what the unexpected behaviour was, and when it occurred.

**What happens next:**

> Reports will be treated in confidence. The priority is to assess and contain any potential data exposure — not to attribute blame. Early reporting significantly reduces the risk and scope of harm. Late or concealed reporting makes both worse.
